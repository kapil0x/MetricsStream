🎯 Overview

Phase 0 is your entry point into distributed systems engineering. Instead of jumping straight into optimizations, you'll build a working monitoring platform with all five core components: ingestion, queuing, storage, querying, and alerting. The system is deliberately simple—single-threaded ingestion, blocking I/O, no indexes—to show you the architecture first, and the bottlenecks second.

🧠 Learning Philosophy

Working first, optimized later. You'll build a complete system in hours, see where it's slow, then spend Crafts #1-5 making each component production-grade. This mirrors how senior engineers work: validate architecture before investing in optimization.

By the end, you'll understand how metrics flow from clients → ingestion → queue → storage → query → alerts, and you'll have measured the performance bottlenecks yourself.

🏗️ Architecture

┌─────────────┐ │ Clients │ (send metrics via HTTP POST) └──────┬──────┘ │ POST /metrics ▼ ┌─────────────────────┐ │ Component 1: │ │ Ingestion API │ (HTTP server, single-threaded, blocking) └──────┬──────────────┘ │ push ▼ ┌─────────────────────┐ │ Component 2: │ │ In-Memory Queue │ (std::queue + mutex, unbounded) └──────┬──────────────┘ │ pop ▼ ┌─────────────────────┐ │ Component 3: │ │ Storage Consumer │ (background thread, blocking file writes) └──────┬──────────────┘ │ write ▼ phase0_metrics.jsonl (JSON Lines format) │ ├──────────────────┐ ▼ ▼ ┌─────────────┐ ┌─────────────┐ │ Component 4:│ │ Component 5:│ │ Query API │ │ Alerting │ └─────────────┘ └─────────────┘ GET /query (polling every 10s)

The 5 Components

1. Ingestion API

HTTP POST /metrics endpoint accepting JSON payloads. Single-threaded, blocking request handling.

Tech: Raw sockets, simple HTTP parsing

2. In-Memory Queue

Thread-safe queue buffering metrics between ingestion and storage. Unbounded, no persistence.

Tech: std::queue + std::mutex

3. Storage Consumer

Background thread polling queue and writing metrics to file. Blocking I/O, append-only.

Tech: std::thread, std::ofstream

4. Query API

HTTP GET /query endpoint reading from file and filtering by name and time range. Full file scan.

Tech: Linear scan, in-memory filtering

5. Alerting Engine

Background thread checking alert rules periodically. Calculates averages, compares to thresholds.

Tech: Polling model, console logging

🚀 Getting Started

1. Build

cd phase0 ./build.sh

2. Run Server

cd build ./phase0_poc

You should see all 5 components start:

[Storage] Consumer started, writing to phase0_metrics.jsonl [Alerting] Added rule: cpu_usage > 80 (window: 60s) [Alerting] Engine started, checking every 10 seconds [Ingestion] Server started on port 8080

3. Test It

Open another terminal and send a metric:

curl -X POST http://localhost:8080/metrics \ -H "Content-Type: application/json" \ -d '{"name":"cpu_usage","value":85}' # Response: {"status":"accepted"}

Query it back:

curl "http://localhost:8080/query?name=cpu_usage&start=0&end=9999999999999" # Response: [{"name":"cpu_usage","value":85,"timestamp":1696789234567}]

🎓 What You'll Learn

Core Concepts

  • Component Boundaries: How to design clean APIs between system components
  • Producer-Consumer Pattern: Thread-safe queues for decoupling producers from consumers
  • HTTP API Design: Building RESTful endpoints for metrics ingestion and querying
  • Time-Series Storage: JSON Lines format for append-only metric storage
  • Rule-Based Alerting: Evaluating thresholds over time windows

Performance Understanding

  • Bottleneck Identification: See single-threaded ingestion limiting throughput to ~50 RPS
  • Blocking I/O Impact: Understand how file writes create backpressure
  • Linear Scan Costs: Feel the slowdown of O(n) queries without indexes
  • Polling Latency: Experience 10-second alert delays from fixed check intervals

⚠️ Intentional Bottlenecks

This system is deliberately slow. The single-threaded ingestion, blocking I/O, and full file scans are educational bottlenecks. You'll feel the pain, measure it with benchmarks, then fix it in Crafts #1-5. This is the learning method.

🔬 Hands-On Learning

Exercises (8-10 hours total)

Deepen your understanding with 10 progressive exercises:

  1. Add Request Logging: Track latency for every request (20 min)
  2. Queue Depth Monitoring: Detect backpressure in real-time (25 min)
  3. Value Range Filtering: Query metrics by value range (30 min)
  4. Batched Ingestion: Accept multiple metrics in one request (45 min)
  5. Alert Deduplication: Prevent repeated alerts for same condition (40 min)
  6. Percentile Queries: Calculate p50, p90, p99 (60 min)
  7. Exponential Backoff: Retry failed storage writes (45 min)
  8. Load Tester: Build a custom benchmarking tool (60 min)
  9. Metric Expiration: Auto-delete old data (50 min)
  10. Web Dashboard: Real-time metrics visualization (90 min)

Performance Benchmarks

Measure system performance with included tools:

./build/simple_benchmark localhost 8080 50 100 # 50 concurrent clients, 100 requests each # Outputs: RPS, p50/p90/p99 latency, success rate

📈 Expected Results

  • Throughput: 50-100 RPS (single-threaded bottleneck)
  • Latency: 10-20ms average, 50-100ms p99
  • Query Time: O(n) scaling - 5ms for 100 metrics, 50ms for 1000
  • Alert Latency: Up to 10 seconds (polling interval)
